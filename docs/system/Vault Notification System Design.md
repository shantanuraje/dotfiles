# Vault Notification System — Design Proposal

Generated by parallel research agents on 2026-04-27. Two passes were run; the
second had read access to the live vault and agent definitions and produced
the more grounded plan. Both are preserved below for reference.

> **Status**: design only, not yet implemented. The `vault-*` topics are
> reserved in `system_scripts/notify/topics.md` but have no publishers yet.

## Critical Findings (verified)

These are real issues to fix *before* wiring vault notifications:

1. **CLAUDE.md scheduled vault tasks are not wired.** No `claude-vault-*.{service,timer}` files exist in `~/.config/systemd/user/`. The 7 scheduled tasks listed in CLAUDE.md ("Scheduled Vault Maintenance Tasks") are aspirational, not running.
2. **Internal CLAUDE.md inconsistency**: the intro says state/wrappers/logs go in `05-Meta/05-03-Scripts/.vault-automation/`, but the prompts redirect logs to `~/.local/share/claude-cron/`. Pick one before wiring.
3. **Stale agent docs**: `vault.md`, `capture.md`, etc. all reference 10 areas. Reality: **7 areas** (Health, Finance, Learning, Career, Personal, Home, Hobbies — no Auto, Productivity, Documents).
4. **Journal hierarchy mismatch**: CLAUDE.md says `06-Journal/YYYY/QX/W-XX/YYYY-MM-DD/`. Reality: `06-Journal/06-02 Daily/YYYY/YYYY-WXX/YYYY-MM-DD/` — no `QX/`, no `W-` prefix. Every agent prompt that calculates journal paths is wrong by one level.
5. **Stray `02-Journal/`** directory at vault root, duplicate of canonical `06-Journal/`. Hygiene cleanup.
6. **Inbox claim is stale**: CLAUDE.md says 12 unprocessed items oldest from 2025-05-30; actual is 2 items only. Backlog has been processed.
7. **`05-03-Scripts/` is not empty** as CLAUDE.md claims — it has `python_task_manager`, `dashboard`, `Kimi-vault-migration-scripts`, `fun`.

**Implication**: reconciling CLAUDE.md + agent .md.tmpl files vs reality is a
prerequisite to wiring the timers, since stale routing rules would push every
scheduled task into wrong paths.

## Architecture (consolidated)

```
┌─ Vault filesystem (~/Documents/personal/) ────────────────────┐
│   YAML frontmatter (TaskNotes), 00-Inbox/, 06-Journal/, …     │
└────────────────────────────────────────────────────────────────┘
       │ read                            ▲ write
       ▼                                 │
┌──────────────────────────┐    ┌──────────────────────────────┐
│ Vault scanners           │    │ Capture sinks                │
│ (systemd user timers)    │    │ - inotifywait → vault-capture│
│ - briefing.sh            │    │ - "ntfy → inbox" listener    │
│ - due-today.sh           │    │   (subscribes to write topic)│
│ - reminder-poll.sh       │    └──────────────────────────────┘
│ - inbox-age.sh           │              ▲ HTTP poll/ws
└────────────┬─────────────┘              │
             │ source notify/lib.sh       │
             ▼                            │
   ntfy server (beelink:8090) ◄───────────┘
             │
             ▼
   Phone / iPad / web UI
```

**Egress** (vault → phone): scanners run on a tailnet host that can read the
vault. They source `notify/lib.sh` and publish to `vault-*` topics. Click
headers use `obsidian://open?vault=personal&file=<encoded-path>` so taps
deep-link into the note.

**Ingress** (phone → vault): a daemon runs `ntfy subscribe …vault-capture-in
--json` and pipes each message into `00-Inbox/YYYY-MM-DD HH-MM <title>.md`
with proper frontmatter. Mobile UX: tap "send" on a topic in the ntfy app,
type a thought, phone POSTs, daemon writes to inbox, capture-confirmation
echoes back on `vault-capture`. SMS-style "text yourself into the vault."

## Per-Task Wiring Plan

When CLAUDE.md tasks are wired, each runs `claude -p "..."`, captures stdout,
sources `lib.sh`, and publishes to a `vault-*` topic.

| CLAUDE.md task | Topic | Severity wrapper | Notes |
|---|---|---|---|
| Morning briefing 07:30 | `vault-briefing` | `notify::warn` if overdue > 0 else `notify::info` | Click → today's daily note URI |
| Work/career 09:00 weekday | `vault-work` | `notify::info` | Click → 02-04-Career active projects |
| Inbox sweep 18:00 | `vault-inbox` | `notify::info` (escalate `notify::warn` if backlog > 10) | |
| Evening review 20:00 | `vault-evening` | `notify::info` | Click → tomorrow's preview note |
| Weekly inbox reminder Sun 10:00 | `vault-inbox` | `notify::warn` | |
| Weekly health Mon 08:00 | `vault-health` | `notify::warn` if issues, `notify::ok` if clean | Full report written under `05-Meta/05-04-System-Management/` and linked via Click |
| Monthly orphan/tag audit | `vault-orphans` / `vault-audit` | `notify::info` | |

All `claude-vault-*.service` units should also have `OnFailure=` pointing to
the existing `errors` topic (via `send-error.sh`).

## Tier Breakdown

**Tier A — pure shell + jq + yq + curl + inotify-tools** (no new deps):
- Inbox count + oldest-age scanner.
- Daily briefing via `find` + `yq -r` + jq sort/count.
- Capture confirmations via `inotifywait -m 00-Inbox/`.
- Health/orphan audits via `grep`/`awk` on frontmatter.
- All scheduled by user systemd timers, all publish via `notify::send`.
- Covers ~80% of use cases. Build first.

**Tier B — adds a Python helper** at `05-Meta/05-03-Scripts/.vault-automation/vault_tasks/`:
- Robust frontmatter parsing via `python-frontmatter`.
- RFC 5545 recurrence expansion via `python-dateutil` (so we know which recurring task is "due today").
- Time-bound reminder polling with proper dedupe state.
- Wikilink graph traversal for orphan detection.
- Needed to fully honor TaskNotes semantics.

**Tier C — Obsidian plugin or external service**:
- Obsidian plugin posting to ntfy on TaskNotes events (task completed, due-today fires) — plugin-internal, not filesystem polling. Custom plugin, ~few hundred LOC. Only worth it if Tier A+B feels too coarse.
- Calendar plugin → ntfy bridge.
- Live `vault-capture-in` ingestion daemon as long-running systemd service.

## Pitfalls

- **Spam risk**: biggest offenders are `vault-due` and `vault-reminders`. Coalesce into morning + evening batches; don't notify per task. Reserve `high` priority for `vault-overdue` only.
- **Timezone**: ISO 8601 frontmatter uses `-04:00`. Recurring tasks across DST boundaries misfire with naive datetime. Use timezone-aware parsing (Python `dateutil` is reliable; `date -d` mostly works on the host).
- **Vault edits at 3am**: AI agent / automation creates daily notes before user wakes. Don't notify on `vault-capture` during 22:00-07:00; gate inotify rule by hour or pass `--quiet` flag from agents.
- **Click-through reality**: `obsidian://open?...` works on desktop reliably. Android Obsidian's URI handler has finicky URL-encoding for nested paths — test against an actual journal note path before standardizing.
- **TaskNotes index vs YAML parse**: TaskNotes maintains its own internal index but doesn't expose it on disk in a stable format. Parse YAML directly to decouple from plugin internals.

## Open Questions for the User

1. **Vault accessible from beelink itself?** Or only the laptop? Affects which host runs the scanners.
2. **Should `vault-overdue` wake the phone?** Or stay iPad-only? (Default suggestion: iPad-only, given your no-spam preference.)
3. **Bidirectional `vault-capture-in` ingestion path** — implement or skip? Powerful, but introduces a write surface.
4. **AI-agent captures**: should `capture.md` / `research.md` emit `vault-capture` notifications by default, or only for `priority: high` items? Default-on may be noisy.

## Recommended Sequence

1. Reconcile CLAUDE.md vs reality (areas count, journal hierarchy, log location). Update agent .md.tmpl files. Decide on log/state location.
2. Build Tier A scanners as wrappers calling `claude -p` + `lib.sh`.
3. Wire 7 CLAUDE.md timers with `OnFailure=` → `errors`.
4. Update `topics.md` "Wired by" column from "(tier 2)" to actual wrapper paths.
5. Address stray `02-Journal/` directory.
6. Evaluate Tier B / Tier C only after Tier A is stable.
